## Recorded as TODOS (not fixed here)

### P1 — Performance: selector storms (UI audit)
The `.filter()`-inside-selector class (twice fixed in Train components) is
systemic: **HomeTab** (`457-458, 472` — default tab re-renders on every store
write), **Dashboard root** (`428` — whole app shell), **ActiveWorkout** (`175`
— 4,600-line in-workout UI), ProgressTab (3 sites), WorkoutHistory/Recovery*/
ReadyForThis/WorkoutStartChooser, plus **31 components using whole-store
`useAppStore()` with no selector** (DietCoach, GrapplingTracker,
SessionTemplates, BodyWeightTracker worst). Fix pattern is established:
select raw refs, derive with useMemo.

### P1 — The UTC date-keying family (data audit: 9 more instances)
Same class as the 3 already fixed. Worst: **wellness XP/streaks**
(`store.ts:3446+`, `gamification.ts:1447` — evening logs credit tomorrow, XP
silently denied next morning), **competition dates** (`daily-directive.ts:177`
— weigh-in/fight-day directives vanish or fire a day early), **workout skips**
(write UTC, read mixed), weekly buckets keyed by non-normalized
`monday.toISOString()` (badges/volume-landmarks/smart-deload/progress-analytics),
streak-shield refill, dual-training-day detection (two paths, two keyings),
nutrition "today" disagreeing across surfaces, water/supplements/login-bonus
day rollover, DST `gap <= 2` streak math. One shared `localDayKey()` helper +
sweep. Note `utils.ts safeDayKey` is itself UTC — fix it first.

### P1 — Brzycki e1RM has no rep cap (data audit)
`weight / (1.0278 - 0.0278*reps)` explodes at high reps and goes NEGATIVE at
37+. Nine call sites; amplified by bodyweight backfill (30-rep push-ups at
80kg BW → e1RM ≈ 430kg → permanent false PR). Shared `calc1RM` with
`reps = min(reps, 12)` or Epley for >10.

### P1 — Dependency vulnerabilities (PWA audit)
`npm audit --omit=dev`: 20 vulns (1 critical: jspdf ≤4.2.0 — non-breaking fix
available). `npm audit fix` covers jspdf/dompurify/lodash/fast-uri now;
schedule `@vercel/postgres` 0.10 and the Next 15/16 migration (next@14.2.35
carries ~14 advisories).

### P1 — skipWaiting() force-reloads users mid-workout on every deploy
`sw.js:21` unconditional `skipWaiting` + `controllerchange→reload`
(page.tsx:137) = random mid-set reloads after deploys; the update banner and
SKIP_WAITING handler are dead code; old-cache deletion causes ChunkLoadErrors
that Sentry is configured to ignore. Let the banner drive activation.

### P2 (selected)
- Push `send` delivers to ANY caller-supplied endpoint under the app's VAPID
  identity (comment claims an ownership check that doesn't exist) — store
  subscriptions server-side
- Subscription tier client-trusted; first-ever sync can persist a self-granted
  `pro` (gap: guard only strips when a server row exists)
- In-memory rate limiter is per-Lambda — ai-coach 3/day not globally enforced
- Whoop tokens stored plaintext if TOKEN_ENCRYPTION_KEY unset (fail closed);
  account deletion doesn't cascade `whoop_tokens`
- SW caches all GET /api/* (incl. auth/session) with no logout purge, no LRU
- Sync queue stores full untrimmed state snapshots in localStorage (quota);
  silently dropped on quota failure; `lastSyncAt` stamped at flush not queue
  time (stale scalars can beat fresh saves)
- Sentry client: no URL scrubbing — `?userId=` lands in traces
- completeWorkout volume math lacks finite-number guard (NaN poisons lifetime
  stat); pause/complete edge inflates duration
- localStorage `setItem` swallows non-quota errors silently; pre-prune path is
  async (write reordering); partialize trims are silent data loss if sync down
- 37 icon-only buttons missing aria-label; WorkoutHistory unbounded list
- manifest.json missing `id`/`scope`; SW fetches have no timeout
- Queue reorder doesn't survive sync (PlannedMesocycle has no position field)
- 30-day tombstone GC vs >30-day-offline device resurrection (documented tradeoff)

### Sync-review residuals (accepted, documented)
- XP LWW drops concurrent increments by design (CRDT counter would be needed);
  displayed XP ratchet (`max(stored, computed)`) can mask reductions
- Offline edit of a completed block can re-activate it (data preserved, nothing
  erased) — rare, structurally indistinguishable from undo without `_revivedAt`
